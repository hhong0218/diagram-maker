How to Make a Flowchart — Symbols, Steps and Examples

Diagram Maker Guide · June 18, 2026

A flowchart is the most common diagram for showing how something works, step by step, using shapes and arrows. You'll find flowcharts in work manuals, system designs, onboarding docs, and presentations. But the moment you sit down to draw one, the basics trip people up: "When do I use a diamond?", "Where does the arrow connect?", "How many endings should there be?" This guide walks you through everything from what each symbol means to the order you should draw in, a worked example, and the mistakes beginners make most often. You won't need to install any software or sign up — you can draw a clean, standard flowchart right in your browser using nothing but shapes and connectors.

1. The flowchart symbols you actually need

The international standard (ISO 5807) defines dozens of symbols, but in practice you only need about six — and the shape panel in Diagram Maker is built around exactly these six.

SymbolNameUse
Rounded rectangleTerminatorThe start and end of a process. Every flowchart should have exactly one start and at least one clear end.
RectangleProcessA normal step of work, such as "Fill out the form" or "Request approval" — one action per box.
DiamondDecisionA yes/no branch. A diamond must have two or more outgoing arrows, each labeled "Yes" or "No".
ParallelogramInput/OutputA point where data enters or leaves, like "Enter customer info" or "Print report".
CircleConnectorA reference point that breaks and rejoins a flow when lines would otherwise cross.
Document shapeDocumentUsed when the output is specifically a paper or report.

2. The order to draw in — flow first, shapes second

A good flowchart isn't one with pretty shapes; it's one with a clear flow. Here's the recommended order.

  1. Write the steps out as text first. Listing "Receive order → Check stock → If in stock, ship; if not, reorder → Deliver" one line at a time quickly reveals missing steps and branches.
  2. Place the start and end as rounded rectangles. If you find yourself with several endings, that's a sign the scope is fuzzy — redefine where the process begins and ends.
  3. Lay process steps out top to bottom. Keep a single direction (top→bottom or left→right) so the reader's eye follows the same path as the logic.
  4. Add decision branches as diamonds and label them. In Diagram Maker, double-click a connector (double-tap on mobile) to type a branch label like "Yes" or "No" directly onto the line.
  5. Trace the whole thing once to verify. Follow every branch to confirm it reaches an end, and check that no arrows are left dangling.

3. How to draw it in Diagram Maker

Diagram Maker keeps the toolbar deliberately small so you can focus on the flow. To create a node, pick a shape in the left panel and double-click an empty spot on the canvas (double-tap on mobile). To connect two nodes, drag from the blue connection point on the edge of a node to another node — a connector is drawn automatically. Double-click any node to edit its text, and double-click a connector to add a label such as "Yes" or "No". You can change a shape's fill, border, and line color in the Selection Style panel, and set the line type to Elbow, Straight, or Curved and the arrow to one-way, two-way, or none. Turn on the grid to keep things aligned, then use "Auto-arrange Nodes" to even out the spacing and "Fit to View" to check the whole chart at once. The fastest way to start is to load the Approval Process template from the right panel and edit it to match your own process.

4. Common mistakes to avoid

First, packing two actions into one box. "Fill out the form and request approval" should be two steps, so that when you look for a bottleneck you can tell which step is the problem. Second, leaving branch labels off. If the lines leaving a diamond have no "Yes"/"No", every reader interprets them differently. Third, mixing arrow directions. If the flow goes top to bottom and then suddenly doubles back upward, the eye loses its place — route loop-backs (like rework) cleanly around one side. Fourth, too large a scope. If a single chart needs more than about thirty shapes, split it into a high-level process and sub-processes so each one stays readable.

5. A worked example — a leave approval process

Here's how a real leave-approval flow looks in practice. Start (rounded rectangle) → "Submit leave request" (rectangle) → "Any leave days left?" (diamond) — if No, "Reject request" (rectangle) then End; if Yes, "Manager approves?" (diamond) — if No, reject with a reason, if Yes, "Record in HR" (rectangle) → "Notify result" (parallelogram) → End. When every branch has a clear outcome like this, even a brand-new employee can understand the whole process from a single page. The same structure applies to purchase approvals, customer refunds, incident response, and almost any approval-style workflow.

Flowchart vs. other diagrams — which to use

SituationRecommended diagram

Showing what to choose under which conditionDecision tree (a flowchart variant)
Branching ideas out from one topicMind map
